# MVP_SCOPE

## MVP 目标

MVP 的目标是完成一个最小可运行、可测试、可回滚的 SleepAgent 原型闭环。它不追求完整产品形态，也不追求呼吸暂停模型立即达到最终性能指标，而是先建立清晰的数据结构、推理流程、评估方式和报告输出骨架。

## MVP 包含内容

### 1. 项目骨架

- 只在 `sleepagent/` 下开发 SleepAgent 项目文件。
- `yasa/` 仅作为第三方源码参考或依赖。
- 建立基础文档、任务日志和增量开发规范。

### 2. 数据接口设计

- 定义 SHHS PSG 数据输入约定。
- 定义睡眠阶段标签映射。
- 定义呼吸事件标签映射。
- 定义单条睡眠记录的统一分析结果结构。

### 3. 睡眠分期 MVP

- 使用 YASA 作为睡眠分期基础模型。
- 将输出简化为 Wake、REM、NREM。
- 计算基础评估指标：
  - Accuracy。
  - Cohen's Kappa。
  - macro F1。
  - weighted F1。
  - 各类别 F1。

### 4. 呼吸事件检测 MVP

- 建立 1D-CNN + BiLSTM 模型骨架。
- 支持 normal breathing、hypopnea、suspected apnea 三分类。
- 明确输入窗口长度、采样率处理和标签对齐策略。
- 提供 smoke test 级别训练或推理流程。
- 输出 Recall、AUC 和 F1。

### 5. 报告 MVP

- 基于结构化分析结果生成两个版本报告：
  - 老人易懂版。
  - 子女/医生专业版。
- MVP 阶段可先使用模板报告。
- RAG 和医学知识库可先设计接口，后续逐步接入 Chroma。

### 6. 可视化 MVP

- 展示睡眠总览。
- 展示 AHI 指数。
- 展示疑似呼吸暂停统计。
- 展示呼吸频率趋势。
- 展示睡眠风险等级。

### 7. 最小服务接口

- FastAPI 提供分析入口。
- Streamlit 提供结果展示入口。
- 初期可使用 mock 数据完成前后端联调。

## MVP 不包含内容

- 不在第一阶段实现完整账号体系。
- 不在第一阶段接入真实推送渠道。
- 不在第一阶段实现完整 PostgreSQL 生产级 schema。
- 不在第一阶段接入真实天气、饮食和生活方式 API。
- 不在第一阶段承诺呼吸暂停模型达到 AUC > 0.85 和 Recall > 0.80。
- 不在第一阶段构建完整医疗诊断流程。

## 验收标准

MVP 被认为完成时，应满足以下条件：

- 可以对一条示例或 mock 睡眠记录运行分析流程。
- 可以输出 Wake、REM、NREM 睡眠分期结果或等价占位结果。
- 可以输出呼吸事件检测结果或模型骨架的可运行结果。
- 可以计算并展示核心指标。
- 可以生成老人易懂版和专业版报告。
- 可以在前端看到核心摘要和至少一个趋势图。
- 每个完成任务都有 `TASK_LOG.md` 记录。

## 增量开发原则

- 每次只实现一个小模块。
- 每次提交前保证可运行、可测试、可回滚。
- 优先实现稳定接口，再替换内部模型或服务。
- 优先写清楚输入输出，再堆复杂逻辑。
- 任何新增模块都需要同时补充最小测试或验证方式。
- 每次任务结束后必须更新 `TASK_LOG.md`。

## 推荐任务顺序

当前开发顺序以 `docs/DEVELOPMENT_ROADMAP.md` 为准。每个对话窗口只推进一个阶段。

1. 阶段 0：项目骨架 + mock 前后端闭环。状态：已完成。
2. 阶段 1：数据结构、mock 数据、指标模块规范化。状态：已完成。
3. 阶段 2：SHHS 数据理解与数据预处理管线。状态：已完成。
4. 阶段 3：YASA 睡眠分期复现与接入。状态：已完成。
5. 阶段 4：1D-CNN + BiLSTM 呼吸暂停模型骨架。状态：已完成。
6. 阶段 5：真实 SHHS 呼吸事件训练数据构建。状态：已完成。
7. 阶段 6：1D-CNN + BiLSTM 训练、评估、推理。状态：已完成。
8. 阶段 7：报告生成模块升级：模板到 RAG。状态：已完成。
9. 阶段 8：Agent 编排：睡眠分析 Agent、报告 Agent、对话 Agent。状态：已完成。
10. 阶段 9：数据管理、长期记忆、报警推送、外部工具。
11. 阶段 10：最终整合、Docker、README、演示脚本、论文/答辩材料。
